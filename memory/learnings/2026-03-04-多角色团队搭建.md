## 🎯 三种协作模式

### 模式 A：主脑 + 专才
- 主脑负责统一调度和派发任务
- 专才互相独立
- 适合：需要严格流程控制的流水线作业

### 模式 B：独立共享
- 没有主脑
- 专才直接面向用户并共享记忆
- 适合：任务边界清晰，用户明确知道该找谁

### 模式 C：混合推荐（推荐！）
- 主脑负责调度
- 专才之间共享记忆
- 用户可绕过主脑直接联系专才
- 兼顾前两者优点，避免单点故障

## 🏗️ 核心架构

### 目录结构原则
- **专才们的 workspace 指向同一个目录**（共享记忆）
- **但 agentDir 各自独立**（独立身份）

### 示例目录结构
`
~/.openclaw/
├── workspace-main/           # 主脑独立 workspace
├── workspace-team-a/         # 团队 A 专才共享 workspace（模式 B/C 使用）
├── agents/
│   ├── main/
│   │   ├── agent/            # 主脑 agentDir（auth、配置）
│   │   └── sessions/
│   ├── coder/
│   │   ├── agent/
│   │   └── sessions/
│   ├── researcher/
│   │   ├── agent/
│   │   └── sessions/
│   └── writer/
│       ├── agent/
│       └── sessions/
└── openclaw.json
`

## 👥 角色配置示例

### 主脑 SOUL.md
`markdown
# SOUL.md - 主脑
你是团队的协调者和对外入口。

## 职责
- 理解用户意图，判断任务复杂度
- 简单任务直接自己处理，不必每次派发
- 编码类 → 派给 coder（sessions_send）
- 调研/搜索类 → 派给 researcher
- 写作/文案类 → 派给 writer
- 收集专才结果，整合后回复用户（不是简单转发）

## 原则
- 派发任务时说清楚背景和期望输出格式
- 等专才完成后再整合，不要催
- 汇总时做真正的整合，给用户一个完整的答案
`

### 专才 SOUL.md
`markdown
# SOUL.md - 专才
你是团队的专业成员，可能被主脑派发任务，也可能被用户直接联系。

## 共享记忆规范
- 读 MEMORY.md 了解团队整体上下文
- 写记忆时标明身份，格式：[coder] 今天完成了...
- 不要覆盖其他 agent 写的记忆条目

## 原则
- 专注自己的领域
- 任务完成后，清晰地返回结果
- 感知到其他专才已经做过的工作，避免重复
`

## ⚙️ openclaw.json 配置要点

### agentToAgent
`json
"tools": {
  "agentToAgent": {
    "enabled": true,
    "allow": ["main", "coder", "researcher", "writer"]
  }
}
`
- allow 列表是双向的
- 专才也可以用 sessions_send 主动给主脑发消息

### dmPolicy
- **主脑**: "pairing" 或 "open"
- **专才（模式 C）**: "allowlist"，只允许你的数字 ID
- **专才（模式 A）**: "disabled"，必须强制通过主脑

### tools.deny
- coder: 可以 deny browser、nodes
- researcher: 可以 deny exec、write、edit
- writer: 可以 deny exec、browser、nodes

## 💡 关键提醒

### 成本提醒
- 一个 business 账号在 5 个小时内对话尽量在 100 次以内
- 想搭建多 Bot 的团队协作，强烈建议多买几个 business 账号或者使用多个不同的模型

### 模式选择建议
| 场景 | 推荐模式 |
|------|----------|
| 刚开始，Bot 较少（2-3 个） | 模式 B，配置简单，上手极快 |
| 任务复杂，需要流程控制 | 模式 A，强管控，适合标准工作流 |
| 生产级、长期稳定运营 | 模式 C（混合），灵活且容错率高 |
| 有多个独立业务线团队 | 各团队内部用模式 C，彻底隔离启动两个独立的 gateway 实例 |

### 并发写记忆
- 偶发，但实际影响小，一般不会同时写同一行
- 建议在写记忆时加上 agent 标识（如 [coder] 修复了xxx问题...）
