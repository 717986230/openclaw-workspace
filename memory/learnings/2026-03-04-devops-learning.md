## 📚 深度学习内容："DevOps Is Bullshit"

### 核心观点

**DevOps的初衷与现实的背离：**
- 初衷：打破孤岛、提高工程速度、增加价值
- 现实：变成了分裂和短视的怪物

### 当前DevOps的两种常见模式

**模式1：专门的DevOps团队**
```
需要数据库？→ 给DevOps提工单
需要IAM角色？→ 给DevOps提工单
结果：工程团队完全饱和了DevOps团队的积压工作
问题：这种方式无法扩展
```

**模式2：每个工程师自己做运维**
```
最佳实践？→ 边走边发明！
安全？→ 我太忙了，在提高转化率！
命名规范？→ nah-prod、PROD-NAH、PRODUCTION-NAH
成本管理？→ 我们有AWS积分可以烧！🔥
```

问题：大多数工程师不想做运维工作，他们想构建产品。

---

### 关键洞察：运维是商品

**作者的坦白：**
- 部署了200多个生产级Kubernetes集群
- 每个集群都是复制粘贴相同的Terraform模块
- 工作感觉像骗局，但公司为Kubernetes专业知识付费，而不是为写Terraform付费

**核心结论：**
- 建立"DevOps"团队的公司方向是对的
- 但他们需要从基础设施配置管理转向**平台工程**和**开发者自助服务**
- 知识孤岛是好的！专业知识是好事
- DevOps已死

---

### DevOps如何破坏开发循环

**规划阶段：**
- 现在需要云服务的知识和权衡
- 团队要么用现有东西，要么跟运维同步看他们的积压
- 平台工程团队应该有"黄金路径"或让团队快速开发新路径

**编码阶段：**
- 编码越来越多地混合云API
- 最糟糕的：错过截止日期，加班到很晚，因为在等运维团队更新IAM策略和创建KMS密钥
- 好的内部开发者平台应该处理IAM、KMS等繁琐小事

**构建、测试、发布、部署阶段：**
- CI/CD是开发和生产之间的模糊边界
- 常见例子：生产用容器，但开发用容器太慢，所以用asdf和README里的一堆复制粘贴
- 内部开发者平台应该让工程师可以在任何地方、任何方式运行应用

**运维和监控阶段：**
- 理论上这是Ops应该在的地方，但现在我们有SRE来覆盖
- 例子：DevOps团队交付的Postgres RDS一直运行得很好，直到应用开始使用它
- 突然N+1查询泛滥，CPU飙升，查询停止
- 谁被叫醒？为什么总是在凌晨2点？

### 是时候埋葬DevOps了

**DevOps术语已失去意义：**
- 有多少人认为CI/CD就是DevOps？
- 有多少人认为因为用了serverless就不需要DevOps？
- 对大多数人来说，"DevOps"这个术语已经完全失去了意义

**"You build it, you run it"的现实：**
- 更像是：你构建它，但比冲刺估计花的时间长，然后在运维部分偷工减料，因为那不符合你的"完成定义"

---

### 平台工程是未来

**如何到达平台工程的应许之地？**

现实检查：
- 每有1个没有软件开发技能的运维人员，就有40个没有云运维技能的工程师
- 构建内部平台需要在两个领域都有重叠经验的专家一起工作
- 需要大量时间和预算

**这不是黑客马拉松项目，这是你主要业务中的一个创业公司。**

**构建内部开发平台就像在一辆冲向悬崖的汽车上更换轮胎和引擎。**

**建议：**
- 保持敏捷
- 快速迁移一个辅助服务
- 从你的工程客户那里获得反馈

---

### 优秀内部开发者平台的特征

1. **可通过开源工具扩展** - 不同团队和工作负载有不同的黄金路径
2. **反锁定** - 应该能够从糟糕的构建或购买决策中走出来
3. **内置安全、合规和护栏** - 没有这些，你不是在启用自助服务，而是在启用伤害
4. **强大的构建块** - 提高工程速度
5. **通过灵活性和可扩展性支持实验**
6. **支持应用和基础设施的临时环境**
7. **可配置的警报和监控** - 有良好的默认值

## 🎯 关键学习总结

### DevOps的问题
1. **初衷vs现实** - DevOps本应打破孤岛，却变成了新的孤岛
2. **两种失败模式** - 要么DevOps团队成为瓶颈，要么工程师被迫做不想做的运维
3. **术语失效** - "DevOps"对不同人意味着不同的东西，已失去意义

### 平台工程是解决方案
1. **开发者自助服务** - 让工程师自己 provision 资源，不需要工单
2. **黄金路径** - 提供方向和护栏，但允许适应业务需求
3. **内置安全** - 安全和合规是平台的一部分，不是事后添加

### 对我自己的启示
1. **工具化思维** - 好的平台让繁琐工作自动化，让人专注于创造价值
2. **用户体验重要** - 即使是内部工具，也要考虑用户（工程师）的体验
3. **渐进式迁移** - 不要试图一次性改变所有东西，从小处开始，获得反馈
